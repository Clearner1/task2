# Task2 API And Export Contract

## Scope

This document defines the first implementation contract for HTTP APIs and exported annotation results.

## Core Resources

- `media`
- `annotation_tasks`
- `annotations`
- `reviews`
- `exports`

## Shared Conventions

### IDs

- `media_id`: use the source filename stem by default, for example `1226-141268-0001`
- `task_id`: use UUID strings generated by the backend
- `annotation_id`: use UUID strings generated by the backend
- `export_batch_id`: use UUID strings generated by the backend

Rules:

- `media_id` must be deterministic for the same source file so repeated scans stay idempotent
- if two different files would produce the same stem, backend must reject the conflict or apply a documented suffixing rule before persistence

### Timestamps

- all API timestamps use ISO 8601 with timezone offset
- example: `2026-04-20T19:30:00+08:00`
- backend must return timezone-aware datetimes only

### Task Status Enum

```text
IMPORTED | PREPROCESSED | READY | IN_PROGRESS | SUBMITTED | REVIEWED | EXPORTED
```

Frontend may mirror this enum exactly in its own type system. The backend contract is the source of truth.

### Pagination Shape

List endpoints use the same pagination response:

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

Rules:

- `page` is 1-based
- default `page_size` is `20`
- maximum `page_size` is `100`

## HTTP Endpoints

### Media

- `GET /api/media`
  - query params: `page`, `page_size`, optional `status`
  - returns paginated media records
- `POST /api/media/import`
  - scans configured input directories
  - registers new media idempotently
- `POST /api/media/preprocess`
  - preprocesses pending media assets
- `GET /api/media/{media_id}`
  - returns stored media metadata and processing state
- `GET /api/media/{media_id}/stream`
  - returns the playable source stream or normalized playable asset
  - supports frontend audio and video playback
  - if byte-range support is implemented, backend should honor range requests for browser media controls

### Annotation Tasks

- `GET /api/tasks`
  - query params: `page`, `page_size`, optional `status`, optional `assigned_to`
  - returns paginated task records
- `GET /api/tasks/next`
  - returns the next available task and acquires a lock
- `GET /api/tasks/{task_id}`
  - returns task details, media references, latest draft view, and latest annotation view
- `POST /api/tasks/{task_id}/autosave`
  - saves draft annotation state and refreshes lock heartbeat
- `POST /api/tasks/{task_id}/heartbeat`
  - refreshes the task lease without writing a new draft annotation
- `POST /api/tasks/{task_id}/release`
  - explicitly releases an in-progress task back to `READY`
- `POST /api/tasks/{task_id}/submit`
  - validates and submits the final annotation payload

### Review And Export

- `POST /api/reviews/{task_id}`
  - records a review decision
- `POST /api/exports`
  - exports a reviewed batch into configured output formats
- `GET /api/exports/{batch_id}`
  - returns export batch metadata and output locations

### Operations

- `GET /api/ops/status`
  - returns operational counters and last maintenance-run state
  - includes task counts, media counts, pending retry jobs, terminal failures, and stale lock count

## Annotation Payload

```json
{
  "primary_emotion": "sad",
  "secondary_emotions": [],
  "intensity": 3,
  "confidence": 4,
  "valence": -0.6,
  "arousal": 2,
  "notes": "语气平稳但整体偏低落"
}
```

Rules:

- `primary_emotion` must come from configured label options
- `secondary_emotions` are optional
- `intensity` and `confidence` use bounded numeric scales
- `valence` and `arousal` are enabled only when configured

## Task Detail Contract

`GET /api/tasks/{task_id}` and `GET /api/tasks/next` return:

```json
{
  "task": {},
  "media": {},
  "latest_draft": null,
  "latest_annotation": null
}
```

Rules:

- `latest_annotation` is the latest persisted annotation for the task, regardless of draft or final state
- `latest_draft` is kept for backward compatibility with current frontend consumers
- after final submit, clients should prefer `latest_annotation`

## Export Contract

### `export-json-contract`

Single-record export shape:

```json
{
  "media_id": "1226-141268-0001",
  "source_path": "task2/media/1226-141268-0001.mp3",
  "media_type": "audio",
  "detected_format": "mp3",
  "duration_ms": 14670,
  "annotation": {
    "primary_emotion": "sad",
    "secondary_emotions": [],
    "intensity": 3,
    "confidence": 4,
    "valence": -0.6,
    "arousal": 2,
    "notes": "语气平稳但整体偏低落"
  },
  "annotator": {
    "annotator_id": "annotator_01",
    "submitted_at": "2026-04-20T19:30:00+08:00"
  },
  "review": {
    "status": "approved",
    "reviewer_id": "reviewer_01",
    "reviewed_at": "2026-04-20T19:35:00+08:00"
  }
}
```

Batch exports may be emitted as:

- `json`: one file per batch
- `jsonl`: one record per line

## Error Mapping

- validation failures -> `400`
- missing records -> `404`
- lock conflicts -> `409`
- unsupported media -> `422`
- transient infrastructure failure after exhausted retries -> `503`

All error responses should include:

- stable error code
- human-readable message
- entity id when available
- retryability hint
